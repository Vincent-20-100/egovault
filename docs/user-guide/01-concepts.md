# 01 ‚Äî Concepts

EgoVault has eight load-bearing concepts. Master these and the rest of the
system reads itself.

## The eight terms

| Term | What it is | Where it lives |
|---|---|---|
| **Source** | A raw ingested artifact (YouTube video, audio file, PDF, web page, raw text). **Immutable** after ingest. Identified by `uid` (UUID4) and a human-readable `slug`. | `sources` table |
| **Chunk** | A `~800` token slice of a source's transcript, used as the unit of RAG retrieval. One source ‚Üí many chunks. Embedded for cosine search; also indexed by FTS5 for BM25. | `chunks` + `chunks_vec` + `chunks_fts` |
| **Note** | Your distilled, human-validated knowledge document. One Markdown file per note in the vault. Has a `title`, `docstring` (one-paragraph summary), `body` (full Markdown), `tags`, `note_type`. | `notes` + `notes_vec` + `notes_fts` + `vault/notes/<slug>.md` |
| **Tag** | A `kebab-case`, lowercase, ASCII-only label. French content, but **no accents**. Notes have a `min_length=1, max_length=10` tag list. | `tags` + `note_tags` |
| **Embedding** | A unit-normalized float vector (768 dims by default with `nomic-embed-text`). Cosine distance ‚àà [0,2]; smaller = closer. | `chunks_vec.embedding`, `notes_vec.embedding` |
| **Run** | One end-to-end execution of a workflow (e.g., one `ingest` call). Identified by `run_id` (UUID4). Used for monitoring, cost tracking, debugging. | `workflow_runs` in `.system.db` |
| **Taxonomy** | User-configurable lists in `system.yaml`: `note_types`, `source_types`, `generation_templates`. *Not engine constraints* ‚Äî the engine validates against the configured lists. | `system.yaml` `taxonomy:` |
| **Template** | A `.yaml` file defining a prompt + output schema used by the LLM to generate a note. Lives in `config/templates/generation/<name>.yaml`. The default is `standard.yaml`. | `config/templates/generation/` |

## Lifecycles

### Source

```
raw ‚Üí (fetched media) ‚Üí (transcribed) ‚Üí (chunked) ‚Üí (embedded) ‚Üí rag_ready
                                                                       ‚îÇ
                                                                       ‚ñº
                                                              pending_deletion ‚Üí (hard delete)
                                                                       ‚îÇ
                                                                       ‚îî‚îÄ failed (any stage error)
```

- `raw` ‚Äî row created, no content yet
- `rag_ready` ‚Äî chunks + embeddings persisted, the source is searchable
- `pending_deletion` ‚Äî soft-deleted (recoverable via `restore_source`)
- `failed` ‚Äî workflow errored mid-way; transcript may be partially saved

A source's text is **never** modified after `rag_ready`. To change it, delete
and re-ingest.

### Note

```
draft ‚Üí active
   ‚îÇ       ‚îÇ
   ‚îú‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚îÄ‚î¥‚îÄ‚îÄ‚îÄ pending_deletion ‚Üí (hard delete)
   ‚îÇ
   ‚îî‚îÄ (no other states; the LLM never edits an active note)
```

- `draft` ‚Äî generated by the LLM, **not yet user-approved**. Not exported to Obsidian.
- `active` ‚Äî user approved the draft. Visible in the vault as a Markdown file.
- Approval normalizes/validates the taxonomy fields (`note_type`, `source_type`)
  against `system.yaml`. The generation step intentionally does NOT enforce
  taxonomy ‚Äî see [04-providers.md](04-providers.md) for why.

## The two-tier knowledge model

EgoVault rejects the standard RAG mental model ("retrieve top-K chunks, paste
into prompt, forget"). It runs **two tiers in parallel**:

```
                       ‚îå‚îÄ‚îÄ‚îÄ‚îÄ Tier 1: RAG on chunks ‚îÄ‚îÄ‚îÄ‚îÄ precise verbatim, raw transcripts
your query ‚îÄ‚îÄ‚ñ∫ curate()‚îÇ
                       ‚îî‚îÄ‚îÄ‚îÄ‚îÄ Tier 2: Compiled notes ‚îÄ‚îÄ dense, human-titled, cross-source synthesis
```

- **Tier 1** is what classical RAG does: the source transcripts split into
  chunks, embedded, semantically searched. Good for exact facts (a specific
  number, a quote).
- **Tier 2** is what the *Knowledge Compiler* thesis adds: notes are crisp,
  reusable, validated knowledge units. They embed and rank far better than
  fragmented chunks (proven on real corpus: top-1 cosine distances ~0.10‚Äì0.13
  lower at tier 2 vs tier 1, see `.meta/audits/2026-05-20-real-notegen-test-results.md`).
- `curate()` orchestrates both. By default it tries notes first, escalates to
  chunks only if notes are insufficient (see [06-search-and-curate.md](06-search-and-curate.md)).

This is *not* generic RAG ‚Äî it's a deliberate move toward an **accumulating,
densifying** knowledge store. See `docs/VISION-KNOWLEDGE-COMPILER.md` for the
full thesis.

## The Librarian ‚Äî `curate()`

A small, **deterministic** tool that exposes the two-tier model to the rest of
the system (CLI, MCP, future API):

```
curate(query, filters?, limit?) -> CuratedContext
                                    ‚îú‚îÄ‚îÄ synthesis  : str   (assembled block of top results)
                                    ‚îú‚îÄ‚îÄ sources    : list  (each with tier, uid, title, content, distance)
                                    ‚îú‚îÄ‚îÄ confidence : float | None  (None at tier 0 ‚Äî no LLM)
                                    ‚îî‚îÄ‚îÄ query      : str
```

**Tier 0** (shipped, current): pure deterministic ‚Äî embed query, search notes,
escalate to chunks if sparse, assemble a minimal context block. No LLM.

**Tier 1** (future, brainstorm open): add an isolated LLM call that selects,
deduplicates, and synthesizes the tier-0 result. Currently open redesign
question ‚Äî see SESSION-CONTEXT open question #7 and the 2026-05-19 SOTA
synthesis card.

## Three storage locations

```
egovault/        ‚Üê public git repo (this code)
egovault-user/   ‚Üê local directory, your data
‚îú‚îÄ‚îÄ data/        ‚Üê vault.db (SQLite) + .system.db + media/ (audio/video downloads)
‚îî‚îÄ‚îÄ vault/       ‚Üê private git repo (Obsidian notes as Markdown)
```

- `data/` is binary; back it up by copying `vault.db`.
- `vault/` is text; back it up via `git push` to a private remote.
- The code repo (`egovault/`) is independent ‚Äî your data never lives in it.
