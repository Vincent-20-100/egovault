## 1. Nomenclature

### 1.1 Language rules

| Context | Language | Examples |
|---|---|---|
| Source code (variables, functions, classes, files) | **English** | `create_note`, `SourceType`, `chunk_size` |
| SQL (tables, columns, indexes) | **English** | `note_uid`, `date_created`, `source_type` |
| Comments and docstrings | **English** | `# Generate UUID4 for new note` |
| Config keys | **English** | `embedding.provider`, `chunking.size` |
| Vault content (notes, tags, slugs) | **French** (default, configurable) | `decentralisation`, `elasticite-prix.md` |

**No exceptions.** When in doubt: English.

### 1.2 Casing conventions

```
Python
в”њв”Ђв”Ђ snake_case     в†’ variables, functions, modules, file names
в”њв”Ђв”Ђ PascalCase     в†’ classes, Pydantic models
в””в”Ђв”Ђ SCREAMING_SNAKE в†’ constants (e.g. DEFAULT_CHUNK_SIZE)

SQL
в””в”Ђв”Ђ snake_case     в†’ tables, columns, indexes, constraints

Vault (notes)
в””в”Ђв”Ђ kebab-case     в†’ note file names, source slugs (no accents, lowercase)
```

### 1.3 Glossary

Each term has exactly one definition. Never use interchangeably.

| Term | Definition |
|---|---|
| `source` | Raw ingested media (YouTube URL, audio file, PDF). Entry point. |
| `note` | Generated Markdown document вЂ” unit of knowledge. System output. |
| `chunk` | Text fragment (~800 tokens) produced by splitting a transcript. RAG unit. |
| `embedding` | Float vector representing a chunk or note in semantic space. Stored in sqlite-vec. |
| `tool` | Atomic Python function: one typed input в†’ one typed output, zero implicit side-effects. |
| `workflow` | Ordered sequence of tool calls implementing a business process (e.g. `ingest_audio`). |
| `uid` | UUID4 generated by the system. Immutable after creation. Primary key everywhere. |
| `slug` | Human-readable kebab-case identifier derived from the title. Stable but not a primary key. |
| `transcript` | Complete raw text of a source before splitting. Stored as TEXT in DB. |
| `taxonomy` | User-configurable classification values (`note_type`, `source_type`, `generation_template`). Not engine contracts. |
| `generation_template` | Named prompt template used to generate a note body. Defines LLM instructions + Markdown structure. Not editable via frontmatter or by the LLM. Modifiable only via a human-triggered regeneration workflow. NULL for manually written notes. |
| `rating` | Integer 1вЂ“5 set exclusively by the user. Intentionally open semantics (importance, quality, priority вЂ” the user decides). Never inferred by the LLM. |
| `run_id` | UUID4 identifying a workflow execution or MCP session. Propagated via `contextvars`. |

## 2. Repo structure

### 2.1 Three distinct locations

```
egovault/                          в†ђ PUBLIC git repo (engine, code, scripts)
egovault-user/                     в†ђ LOCAL directory (not a git repo)
в”њв”Ђв”Ђ data/                          в†ђ structured data + binary files
в””в”Ђв”Ђ vault/                         в†ђ PRIVATE git repo (Obsidian notes)
```

**Rationale:**
- Code and data have different backup strategies, audiences, and cadences.
- Backup `data/` = copy `vault.db`. No git needed.
- Backup `vault/` = `git push` to private GitHub. Lightweight text, readable anywhere.
- Future cloud migration: SQLite в†’ PostgreSQL + media в†’ S3. `vault/` becomes optional.

### 2.3 Dependency rules (strict) вЂ” VaultContext pattern

```
surfaces (CLI, MCP server, api/)
    в”‚  build_context() в†’ VaultContext
    в†“
tools/ and workflows/          в†ђ receive ctx: VaultContext, import core/ only
    в†“
core/                          в†ђ VaultContext, Protocols, schemas вЂ” zero project imports
    в†‘
infrastructure/                в†ђ VaultDB, build_context, providers вЂ” imports core/ only
```

**VaultContext** is the dependency injection container. It holds all infrastructure dependencies
(DB facade, embedding function, LLM function, vault writer) as a single dataclass.

- **Surfaces** call `build_context(settings)` at startup, then pass `ctx` to tools/workflows
- **Tools** receive `ctx: VaultContext` вЂ” they never import infrastructure/ directly
- **core/** defines Protocols (`EmbedFn`, `GenerateFn`, `WriteNoteFn`) вЂ” type-safe callables
- **infrastructure/** implements the Protocols and provides `build_context()` factory

**Hard rules:**
- `tools/` has **zero imports** from `infrastructure/` (verified by grep)
- `core/` has **zero runtime imports** from `infrastructure/` (TYPE_CHECKING only for VaultDB type hint)
- A tool **never imports another tool** вЂ” if that need arises, the boundary is wrong
- Surfaces contain **zero business logic** вЂ” routing only

**Consequence:** replacing SQLite with PostgreSQL = change `infrastructure/db.py` + `VaultDB`. Zero changes in `core/` or `tools/`. Swapping the embedding provider = change `build_context()`. Tools never know.

## 3. Configuration

### 3.1 Three config files

| File | Versioned | Owner | Role |
|---|---|---|---|
| `config/system.yaml` | Yes (in repo) | Developer | Algo params, taxonomy defaults |
| `config/user.yaml` | No (gitignored) | User | Preferences (embedding model, LLM, language) |
| `config/install.yaml` | No (gitignored) | Machine | Paths, API keys, service URLs |

`core/config.py` loads all three, merges them, and validates them via a single Pydantic `Settings` model at startup. Missing required fields в†’ explicit error before any execution.

### 3.2 system.yaml (algo params + taxonomy)

```yaml
# ============================================================
# SYSTEM CONFIGURATION вЂ” versioned, shipped with the app
# Modify with care: changes affect all users
# ============================================================

chunking:
  size: 800          # tokens per chunk
  overlap: 80        # sliding window (10% of size)

llm:
  max_retries: 2     # Pydantic revalidation attempts before human-in-the-loop
  large_format_threshold_tokens: 50000
                     # Sources exceeding this threshold are indexed for RAG
                     # but note generation is blocked. See large format policy (section 4.1).

embedding:
  dims: 768          # single source of truth for all vec tables (chunks_vec, notes_vec, semantic_cache_vec)
  model: "nomic-embed-text"

# ============================================================
# SEARCH вЂ” reranking + semantic cache
# ============================================================

search:
  reranking:
    enabled: true
    model: "cross-encoder/ms-marco-MiniLM-L-6-v2"   # ~80MB, CPU-compatible
    rerank_factor: 4          # candidates fetched = limit x rerank_factor before reranking
    apply_to_chunks: true
    apply_to_notes: true
  semantic_cache:
    enabled: true
    similarity_threshold: 0.05   # max cosine distance for a semantic hit
    ttl_days: 7
    max_entries: 1000             # FIFO purge (oldest first) if exceeded

# ============================================================
# BENCHMARK вЂ” RAG quality thresholds
# ============================================================

benchmark:
  search:
    min_precision_at_5: 0.8
    min_recall_at_5: 0.7
    min_mrr: 0.6
  chunk:
    max_tokens_per_chunk: 800

# ============================================================
# TAXONOMY вЂ” configurable classification values
#
# IMPORTANT: These are opinionated defaults, NOT engine constraints.
# They reflect a user's workflow. Add, rename, delete freely.
# The engine validates against this list вЂ” it never hardcodes enums.
# Adding a type = one line here, zero code changes.
# ============================================================

taxonomy:
  note_types:
    - synthese        # distillation of an external source + personal perspective
    - concept         # cross-cutting hub note linking multiple sources
    - reflexion       # personal thought, no external source
    - idee            # raw capture, undeveloped starting point

  source_types:
    - youtube
    - audio
    - video
    - pdf
    - livre           # same pipeline as pdf, different semantic context
    - web             # future work item вЂ” not yet implemented
    - personnel       # no external source, personal reflection

  # ============================================================
  # GENERATION TEMPLATES вЂ” names of available prompt templates
  #
  # NOTE: note_type answers "WHAT IS this note?" (content classification, editable).
  #        generation_template answers "HOW WAS this note MADE?" (process trace, immutable).
  #        They may have similar names but are different things.
  #
  # Each name must correspond to a .yaml file in config/templates/generation/
  # Adding a template = one line here + one .yaml file. Zero code changes.
  # ============================================================
  generation_templates:
    - standard        # default: balanced extraction, shipped with the repo
```

## 8. Development guidelines

### 8.1 Modularity as a constraint

> If modifying X requires modifying Y, the boundary is drawn incorrectly.

- Each tool: one responsibility, one typed input, one typed output
- No cross-imports between tools
- Workflows orchestrate, they contain no business logic
- Every "magic value" (threshold, size, timeout) lives in `config/system.yaml`
- No hardcoded paths in code

### 8.2 Living documentation (mandatory)

Any change to behavior, schema, config, or contracts в†’ update `docs/architecture/ARCHITECTURE.md` immediately.

| Type of change | Files to update |
|---|---|
| New tool or workflow | `docs/architecture/ARCHITECTURE.md`, `mcp/server.py` |
| Pydantic schema change | `core/schemas.py`, `docs/architecture/ARCHITECTURE.md` section 6 |
| Config key added/removed | `config/system.yaml`, `docs/architecture/ARCHITECTURE.md` section 3 |
| vault.db schema change | Migration script + `docs/architecture/ARCHITECTURE.md` section 4 |
| .system.db schema change | Migration script + `docs/architecture/ARCHITECTURE.md` section 5 |

**This rule applies to Claude as much as to the developer.**

### 8.3 Monitoring via `@loggable`

```python
@loggable("transcribe")
def transcribe(input: TranscribeInput) -> TranscriptResult: ...
```

After implementing the monitoring spec: `run_id` read from `contextvars`, `token_count` automatically extracted.

Useful SQL queries:
```sql
SELECT tool_name, AVG(duration_ms), COUNT(*) FROM tool_logs GROUP BY tool_name;
SELECT * FROM tool_logs WHERE status = 'failed' ORDER BY timestamp DESC LIMIT 20;
SELECT wr.workflow, SUM(tl.token_count) FROM workflow_runs wr
  JOIN tool_logs tl ON tl.run_id = wr.run_id GROUP BY wr.workflow;
```

### 8.4 Taxonomy extensibility rule

> `taxonomy.*` in `system.yaml` are user opinions, not engine contracts.
> Never hardcode allowed values as Python `Literal` types.

```python
# WRONG
note_type: Literal["synthese", "concept", "reflexion", "idee"]

# RIGHT
note_type: str | None = None
# validator reads config.taxonomy.note_types and checks membership
```

### 8.5 Pre-commit checklist

- [ ] New tool with Pydantic types for input and output
- [ ] New tool decorated with `@loggable`
- [ ] New tool exposed in `mcp/server.py` if user-facing
- [ ] No hardcoded values вЂ” config lives in `system.yaml`
- [ ] No hardcoded taxonomy enums вЂ” validated against `system.yaml:taxonomy`
- [ ] No cross-imports between `tools/`
- [ ] New template: `.yaml` in `config/templates/generation/` + name in `system.yaml:taxonomy.generation_templates`
- [ ] Test added in `tests/` mirroring the tool's path
- [ ] `docs/architecture/ARCHITECTURE.md` updated if the change affects the architecture

## 10. Security Model

**Last updated:** 2026-03-29
**Full spec:** `docs/superpowers/specs/2026-03-29-security-design.md`

### 10.1 Scope вЂ” Local use only

EgoVault is designed for **single-user, local-only** use. The API server listens on `127.0.0.1:8000`. The MCP server is local.

> **CRITICAL WARNING: it is FORBIDDEN to expose EgoVault on any network (LAN, VPN, internet) without a full dedicated security audit covering: authentication, authorization, TLS encryption, CSRF protection, network rate limiting, data isolation, and GDPR/personal data compliance. The current security model provides NO protection against network attacks.**

### 10.2 Sensitive data

| File | Sensitive content | Protection |
|---------|-----------------|------------|
| `config/install.yaml` | API keys (OpenAI, Anthropic) | `.gitignore`, never committed |
| `vault.db` | User notes, sources, embeddings | Restrictive permissions (0600 Unix) |
| `.system.db` | Logs, cache queries, jobs | Restrictive permissions (0600 Unix) |
| `egovault-user/vault/` | Exported Markdown notes | User directory, `.gitignore` |

### 10.3 Documented attack surfaces

- **User inputs**: YouTube URLs, uploaded files (audio/PDF), MCP paths вЂ” validated at entry
- **Logs**: automatic redaction of API keys and system paths before writing to `tool_logs`
- **Database**: CHECK constraint on slug, PRAGMA foreign_keys enabled, parameterized queries
- **API**: rate limiting on expensive endpoints, upload size limit
- **External providers**: if `provider != ollama`, data (text, queries) is sent to an external service вЂ” documented in config files

### 10.4 External providers вЂ” guardrails

When an external provider is configured (OpenAI, Anthropic):
- API keys are **never** logged, exposed in errors, or returned by the API
- The provider used is tracked in each `tool_log` (`provider` field)
- The frontend **never** receives an API key
- An explicit warning is displayed in config files
